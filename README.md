# 프로젝트 개요

이 프로젝트는 Python으로 구현한 콘솔 기반 퀴즈 게임입니다.

사용자는 메뉴를 통해 퀴즈를 풀거나 새로운 퀴즈를 추가할 수 있으며,
저장된 퀴즈 목록을 확인하고 최고 점수를 관리할 수 있습니다.
프로그램은 실행 중 생성되는 데이터를 JSON 파일(state.json)에 저장하여,
재실행 시에도 이전 상태를 유지할 수 있도록 설계했습니다.

본 시스템은 역할 분리와 책임 기반 설계를 기반으로 구현되었습니다.
도메인 모델(Quiz), 저장소(StateRepository), 서비스 계층(QuizGame)을 분리하여
유지보수성과 확장성을 고려한 구조를 갖도록 설계했습니다.

# 퀴즈 주제 선정 이유

본 프로젝트의 퀴즈 주제는 Python 기초 문법과 개념으로 선정했습니다.

Python은 간결하고 직관적인 문법을 가진 프로그래밍 언어로,
프로그래밍 입문자부터 실무 개발자까지 널리 사용되고 있습니다.
따라서 문제의 정답이 명확하고,
난이도 조절이 용이하며,
교육적인 활용 가치가 높다고 판단했습니다.

또한 개발 과제의 구현 언어가 Python이기 때문에,
동일한 언어를 주제로 퀴즈를 구성함으로써
프로그램의 일관성과 학습 효과를 동시에 확보하고자 했습니다.


# 1. 문제 정의 (Problem Statement)

- 사용자가 잘못된 입력을 하면 어떻게 되는가?
- 프로그램이 강제로 종료되면 데이터는 어떻게 되는가?
- 저장 파일이 손상되면 시스템은 어떻게 동작해야 하는가?
- 시스템 상태는 언제, 어떻게 저장되어야 하는가?

이 프로젝트는 이러한 질문에 대해 **명시적인 정책과 동작을 가지는 작은 시스템**을 만드는 것을 목표로 합니다.

---

# 2. 시스템 목표 (System Goals)

이 시스템은 다음과 같은 우선순위를 기반으로 설계되었습니다.

## 핵심 목표

1. 예측 가능한 동작 (Predictable Behavior)
2. 상태 일관성 유지 (State Consistency)
3. 명확한 책임 분리 (Clear Responsibility Boundaries)
4. 실패 대응 가능 (Failure Tolerance)
5. 확장 가능한 구조 (Extensibility)


---

# 3. 시스템 개요 (System Overview)

이 애플리케이션은 상태를 가지는(Stateful) 명령줄 기반 프로그램입니다.

사용자는 다음과 같은 작업을 수행할 수 있습니다.

- 퀴즈 풀기
- 퀴즈 추가
- 퀴즈 목록 조회
- 최고 점수 확인
- 프로그램 종료

모든 실행 상태는 로컬 파일에 저장되며, 프로그램 재시작 시 이전 상태가 복원됩니다.

# 4. 아키텍처 (Architecture)

이 시스템은 **책임 기반 설계(Responsibility-Driven Design)**를 따르며,

각 구성 요소가 명확한 책임을 가지도록 구조화되어 있습니다.

시스템의 목적은 단순한 기능 구현이 아니라,

**예측 가능한 동작과 일관된 상태 관리**를 보장하는 것입니다.

---

## 구성 요소 (Components)

```
QuizGame
├── 프로그램 생명주기 관리
├── 사용자 입력 처리
├── 상태 변경 및 게임 흐름 제어
└── 데이터 저장/불러오기 요청

Quiz
├── 퀴즈 데이터 표현
├── 정답 검증
└── 데이터 무결성 유지
```

---

## 설계 원칙 (Design Principles)

### Single Responsibility Principle (SRP)

각 클래스는 하나의 명확한 책임을 가지도록 설계했습니다.

이를 통해 코드 변경 시 영향 범위를 최소화하고, 시스템의 예측 가능성을 높였습니다.

예:

```
Quiz
→ 도메인 데이터와 규칙을 관리

QuizGame
→ 시스템 동작과 상태 전이를 관리
```

이 구조는 다음을 가능하게 합니다:

- 기능 확장 시 코드 변경 범위 최소화
- 테스트 가능성 향상
- 유지보수 비용 감소

---

# 5. 상태 관리 (State Management)

이 시스템은 실행 중 생성되는 모든 상태를 **단일 상태 저장소(state.json)**를 통해 관리합니다.

상태 관리는 다음 원칙을 따릅니다:

- 상태는 항상 명시적으로 저장된다
- 상태는 프로그램 재시작 시 복원된다
- 상태 변경은 예측 가능한 시점에 발생한다

---

## 관리되는 상태 (Managed State)

```
state.json
```

저장되는 데이터:

- 퀴즈 목록
- 최고 점수

---

## 상태 생명주기 (State Lifecycle)

시스템은 다음과 같은 상태 전이 흐름을 가집니다.

```
프로그램 시작
      ↓
상태 로드
      ↓
게임 실행
      ↓
상태 변경
      ↓
상태 저장
      ↓
안전 종료
```

이 흐름은 다음을 보장합니다:

- 데이터 일관성 유지
- 상태 손실 방지
- 예측 가능한 종료 동작

---

# 6. 데이터 영속성 전략 (Persistence Strategy)

데이터 영속성은 **파일 기반 JSON 직렬화**를 사용하여 구현했습니다.

이 선택은 단순성을 유지하면서도,

**디버깅 가능성과 상태 가시성(Observability)**을 확보하기 위한 결정입니다.

---

## JSON을 선택한 이유

- 사람이 읽을 수 있는 형식 (Human-readable)
- 표준 라이브러리로 지원됨 (No external dependency)
- 장애 발생 시 수동 복구 가능
- 데이터 구조 표현이 명확함

---

## 저장 정책 (Persistence Policy)

시스템은 다음 시점에 상태를 저장합니다.

- 프로그램 종료 시
- 상태가 의미 있게 변경된 시점
- 예외 발생 후 종료 직전

이 정책의 목적은 다음을 보장하는 것입니다:

- 데이터 유실 최소화
- 상태 일관성 유지
- 복구 가능성 확보

---

# 7. 실패 대응 정책 (Failure Handling Policy)

이 시스템은 실패가 발생할 수 있다는 전제로 설계되었습니다.

따라서 시스템은 오류를 숨기지 않고,

명시적으로 감지하고 복구 가능한 상태로 전환합니다.

---

## 처리 대상 실패 (Failure Scenarios)

다음 상황을 명시적으로 처리합니다.

- 상태 파일이 존재하지 않는 경우
- 상태 파일이 손상된 경우
- 파일 쓰기 실패
- 잘못된 사용자 입력
- 프로그램 강제 종료 (KeyboardInterrupt)
- 입력 스트림 종료 (EOFError)

---

## 복구 전략 (Recovery Strategy)

시스템은 오류 발생 시 즉시 종료하지 않고,

안전한 기본 상태로 복구를 시도합니다.

예:

```
파일 손상 감지
      ↓
경고 메시지 출력
      ↓
기본 상태 초기화
      ↓
시스템 계속 실행
```

이 전략의 목표는:

- 서비스 중단 최소화
- 사용자 경험 보호
- 시스템 신뢰성 유지

---

# 8. 입력 검증 전략 (Input Validation Strategy)

사용자 입력은 항상 신뢰할 수 없다고 가정합니다.

따라서 모든 입력은 시스템 내부로 전달되기 전에

검증 및 정규화 과정을 거칩니다.

---

## 검증 규칙 (Validation Rules)

시스템은 다음 조건을 만족하지 않는 입력을 거부합니다.

- 빈 입력
- 숫자 변환 실패
- 허용 범위를 벗어난 값
- 공백 문자열

또한:

> 시스템은 절대 "정상 입력"을 가정하지 않습니다.
> 

---

## 입력 처리 목표

- 시스템 안정성 확보
- 예외 발생 최소화
- 사용자 오류 복구 가능성 제공

---

# 9. 런타임 안정성 (Runtime Safety)

이 시스템은 예상 가능한 종료 동작을 보장하도록 설계되었습니다.

시스템은 다음 상황에서도 비정상 종료하지 않도록 처리합니다.

---

## 처리되는 예외 (Handled Exceptions)

- KeyboardInterrupt
- EOFError
- File I/O Error

---

## 종료 정책 (Shutdown Policy)

프로그램 종료 시 시스템은 다음 순서를 따릅니다.

```
종료 요청 감지
      ↓
현재 상태 저장 시도
      ↓
저장 결과 확인
      ↓
프로그램 종료
```

이 정책은 다음을 보장합니다:

- 데이터 손실 방지
- 상태 일관성 유지
- 안전한 종료

---

# 10. 데이터 모델 (Data Model)

```
Quiz
├── question
├── choices
└── answer
```

이 모델은 시스템의 핵심 도메인 개념을 표현합니다.

---

## 데이터 제약 조건 (Data Constraints)

시스템은 다음 제약 조건을 강제합니다.

- choices는 정확히 4개여야 한다
- answer는 1~4 범위여야 한다

이 제약 조건은 객체 생성 시 검증되며,

유효하지 않은 데이터는 시스템 내부로 전달되지 않습니다.

---

## 데이터 모델 설계 의도

이 모델은 다음을 보장하기 위해 설계되었습니다.

- 데이터 무결성 유지
- 예측 가능한 동작
- 오류 발생 가능성 감소
